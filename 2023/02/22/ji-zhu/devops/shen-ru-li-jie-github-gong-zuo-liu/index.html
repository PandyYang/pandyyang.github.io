关于工作流

工作流是将运行一个或多个作业的可配置自动化过程。工作流由签入到存储库的 YAML 文件定义，并在存储库中的事件触发时运行，也可以手动触发或按定义的计划触发。

工作流在存储库的目录中定义，一个存储库可以有多个工作流，每个工作流可以执行一组不同的任务。例如，您可以有一个工作流来构建和测试拉取请求，另一个工作流在每次创建发布时部署应用程序，还有另一个工作流在每次有人打开新问题时添加标签。.github/workflows

工作流的基础知识

工作流必须包含以下基本组件：

  1. 将触发工作流的一个或多个事件。
  2. 一个或多个作业，每个作业将在运行器计算机上执行并运行一系列一个或多个步骤。
  3. 每个步骤都可以运行您定义的脚本或运行操作，这是一个可重用的扩展，可以简化您的工作流。

触发工作流

工作流触发器是导致工作流运行的事件。这些事件可以是：

  • 工作流存储库中发生的事件
  • 在 GitHub 外部发生的事件，并在 GitHub 上触发事件repository_dispatch
  • 预定时间
  • 手动

例如，您可以将工作流配置为在推送到存储库的默认分支、创建发布或打开问题时运行。

创建工作流

有关工作流的语法，请在目录跳转至对应章节。

GitHub 操作使用 YAML 语法来定义工作流。每个工作流都作为单独的 YAML 文件存储在代码存储库中名为 的目录中。.github/workflows

您可以在存储库中创建一个示例工作流，该工作流在推送代码时自动触发一系列命令。在此工作流中，GitHub Actions 签出推送的代码，安装 bats 测试框架，并运行基本命令来输出 bats 版本：。bats -v

  1. 在存储库中，创建用于存储工作流文件的目录。.github/workflows/
  2. 在目录中，创建一个名为的新文件并添加以下代码。.github/workflows/``learn-github-actions.yml
name: learn-github-actions
run-name: ${{ github.actor }} is learning GitHub Actions
on: [push]
jobs:
  check-bats-version:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '14'
      - run: npm install -g bats
      - run: bats -v
  1. 提交代码并推送至Github存储库。

高级工作流功能

存储机密

如果您的工作流使用敏感数据（例如密码或证书），则可以将其作为机密保存在 GitHub 中，然后在工作流中将其用作环境变量。这意味着您将能够创建和共享工作流，而无需直接在工作流的 YAML 源中嵌入敏感值。

此示例作业演示如何将现有密钥引用为环境变量，并将其作为参数发送到示例命令。

jobs:
  example-job:
    runs-on: ubuntu-latest
    steps:
      - name: Retrieve secret
        env:
          super_secret: ${{ secrets.SUPERSECRET }}
        run: |
          example-command "$super_secret"

创建依赖作业

默认情况下，工作流中的所有作业同时并行运行。如果作业必须在另一个作业完成后运行，则可以使用该关键字创建此依赖项。如果其中一个作业失败，则会跳过所有依赖作业;但是，如果需要继续作业，则可以使用条件语句定义此语句。needs``if

在此示例中，setup``build``test``build``test 和 作业按顺序运行，并且依赖于它们前面的作业的成功完成：

jobs:
  setup:
    runs-on: ubuntu-latest
    steps:
      - run: ./setup_server.sh
  build:
    needs: setup
    runs-on: ubuntu-latest
    steps:
      - run: ./build_server.sh
  test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: ./test_server.sh

使用矩阵

矩阵策略允许您在单个作业定义中使用变量来自动创建基于变量组合的多个作业运行。例如，可以使用矩阵策略在一种语言的多个版本或多个操作系统上测试代码。矩阵是使用关键字strategy创建的，该关键字以数组的形式接收构建选项。例如，此矩阵将使用不同版本的 Node.js 多次运行作业：

jobs:
  build:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [12, 14, 16]
    steps:
      - uses: actions/setup-node@v3
        with:
          node-version: ${{ matrix.node }}

缓存依赖项

如果作业定期重用依赖项，则可以考虑缓存这些文件以帮助提高性能。创建缓存后，它可用于同一存储库中的所有工作流。

此示例演示如何缓存目录： ~/.npm

jobs:
  example-job:
    steps:
      - name: Cache node modules
        uses: actions/cache@v3
        env:
          cache-name: cache-node-modules
        with:
          path: ~/.npm
          key: ${{ runner.os }}-build-${{ env.cache-name }}-${{ hashFiles('**/package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-build-${{ env.cache-name }}-

使用数据库或者服务容器

如果作业需要数据库或缓存服务，则可以使用 services 关键字创建临时容器来托管服务;然后，生成的容器可用于该作业中的所有步骤，并在作业完成后删除。此示例演示作业如何使用它创建容器，然后用于连接到服务。services``postgres``node

jobs:
  container-job:
    runs-on: ubuntu-latest
    container: node:10.18-jessie
    services:
      postgres:
        image: postgres
    steps:
      - name: Check out repository code
        uses: actions/checkout@v3
      - name: Install dependencies
        run: npm ci
      - name: Connect to PostgreSQL
        run: node client.js
        env:
          POSTGRES_HOST: postgres
          POSTGRES_PORT: 5432

使用标签传送工作流

如果要确保特定类型的运行器将处理您的作业，则可以使用标签来控制作业的执行位置。除了默认标签 .然后，您可以在 YAML 工作流中引用这些标签，确保以可预测的方式路由作业。GitHub 托管的运行器分配了预定义的标签。self-hosted

此示例显示工作流如何使用标签来指定所需的运行器：

jobs:
  example-job:
    runs-on: [self-hosted, linux, x64, gpu]

工作流将仅在具有数组中所有标签的运行器上运行。作业将优先转到具有指定标签的空闲自托管运行器。如果没有可用的运行器，并且存在具有指定标签的 GitHub 托管运行器，则作业将转到 GitHub 托管的运行器。runs-on

使用环境

使用作业

工作流运行由一个或多个 jobs 组成，默认情况下并行运行。 若要按顺序运行作业，可以使用 jobs.<job_id>.needs 关键字定义对其他作业的依赖关系。

每个作业在 runs-on 指定的运行器环境中运行。

在工作流程的使用限制之内可运行无限数量的作业。

设置作业的ID

使用 jobs.<job_id> 为作业提供唯一标识符。 键 job_id 是一个字符串，其值是作业配置数据的映射。 必须将 <job_id> 替换为对于 jobs 对象的唯一字符串。 <job_id> 必须以字母或 _ 开头，并且只能包含字母数字字符、- 或 _。

示例：创建作业

在此示例中，已创建两个作业，其 job_id 值为 my_first_job 和 my_second_job。

jobs:
  my_first_job:
    name: My first job
  my_second_job:
    name: My second job

设置作业的名称

使用 jobs.<job_id>.name 设置作业名称，该名称显示在 GitHub UI 中。

定义必备作业

使用 jobs.<job_id>.needs 标识运行此作业之前必须成功完成的所有作业。 它可以是一个字符串，也可以是字符串数组。 如果某个作业失败，则所有需要它的作业都会被跳过，除非这些作业使用让该作业继续的条件表达式。 如果运行包含一系列相互需要的作业，则故障将从故障点开始，应用于依赖项链中的所有作业。

jobs:
  job1:
  job2:
    needs: job1
  job3:
    needs: [job1, job2]

在此示例中，job1 必须在 job2 开始之前成功完成，并且 job3 等待 job1 和 job2 完成。

此示例中的作业按顺序运行：

  1. job1
  2. job2
  3. job3

示例：不要求成功的作业。

jobs:
  job1:
  job2:
    needs: job1
  job3:
    if: ${{ always() }}
    needs: [job1, job2]

在此示例中，job3 使用 always() 条件表达式，确保始终在 job1 和 job2 完成（无论是否成功）后运行。
