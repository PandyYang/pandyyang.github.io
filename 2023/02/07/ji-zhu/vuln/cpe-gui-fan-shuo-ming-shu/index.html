基本格式

CPE Name是一个URI，每个名字都包含前缀 “cpe:”, 最多三个部分。

每个主要的部分都由斜线开头，可能包含0个或多个元素，每个元素是一个已知的硬件和特定平台上软件的组合。多个元素如果在一个部分，必须以分号分割。

下面是CPE Name的基本语法，第一个斜线必须出现，第二个和第三个视他们的期望部分是否为空，来决定是否出现。

cpe:/ {hardware-part} [ / {OS-part} [ / {application-part} ] ]

  • Hardware Facet Specification（硬件方面的规范）

    这一部分定义了it平台方面的硬件规范，其中的每个元素都表示机箱、模块、卡或其他物理指定平台的方面。假如一个CPE名称的硬件部分为空，则表示所有平台。

  • Operating System Facet Specification（操作系统方面的规范）

    这部分定义了it平台方面关于操作系统组成的部分， 每个在os部分的元素描述了一个操作系统运行的平台，对于多核处理器和包含多个操作系统的分布式系统，多个元素可能表明多个操作系统在被使用。假如os部分为空，表示所有的操作系统。

  • Application Environment Facet Specification（应用环境方面的规范）

    这一部分定义了it平台上所关联的应用程序，服务和软件包。每个元素表示一个典型的软件包，如果一个cpe的此部分为空或缺失，则代表所有相关应用程序。

每个部分都由一个或者多个组件组成，中间以冒号分割。第一个组件命名了元素的卖方或供应商，子元素提供了一些额外的信息。3.2版本提供了关于元素和组件的全部语法信息。部分中元素的顺序并不重要； 尽管构成元素的组件的顺序很重要。重复的元素一般不会出现。

示例

单独的cpe名字包含空的软件部分，没有os和app部分，以下表示所有的it系统。

cpe:/

一个典型的cpe名称可能仅包含一个单独的系统平台，硬件部分留空。下面的示例表示Microsofy Windows 2000 操作系统，及其所有的更新和相关版本。

cpe://microsoft:windows:2000

一个更典型的cpe名称可能包含一个操作系统和几个服务器应用，下面的示例是关于RedHat Linux App Server 3，运行于Apache的2.0.52版本的http服务器上。

cpe://redhat:enterprise_linux:3:as/apache:httpd:2.0.52

关于网络设备的cpe名称可能同时包含硬件部分和操作系统部分，下面的示例是关于运行 Internet Operating System™ (IOS) 12.3 版企业版的 Cisco 3825 型集成多业务路由器。

cpe:/cisco::3825/cisco:ios:12.3:enterprise

cpe名称可能仅使用一个应用，下面的示例中os部分和硬件部分为空，表示一个浏览器，没有指定运行于哪种平台上。

cpe:///microsoft:ie:6.0

cpe名称中可以使用感叹号在一个组件中表示多种替代名词，下面的例子表示了一个windows xp系统或者vista。（！就是或的意思）

cpe://microsoft:windows:xp!vista

cpe使用～来指定每个组成仅除了的含义，例如下面的cpe名字定义了微软的windows每种系统除了运行xp。（～就是除了的意思）

cpe://microsoft:windows:~xp

注意or和not操作通常不会一起使用，大多数情况下，它们之一会用于单个组件中。

每个cpe名称可能会包含多个元素，中间以分号分隔，名字部分的元素顺序不重要。元素可能出现在任何顺序位置，下面的例子是关于Sun的Solaris 9 系统同时安装了一个数据库管理器和web服务器。

cpe://sun:sunos:5.9/bea:weblogic:8.1;mysql:server:5.0

元素和组件的语法

CPE 名称的语法受到严格控制，以确保能够维护唯一名称。CPE 的要求之一是只有一种方式来声明特定的平台名称和为此，必须严格执行所使用的确切句法和缩写。 这部分试图解释创建新 CPE 名称时遵循的逻辑。 它的目的是帮助保持新的 CPE 名称遵循一致的命名约定。

语法

cpe名称中的一个元素定义了平台的一部分。每个元素包含一个或多个被冒号分割的组件。第一个元素通常定义了一个供应商的平台，其他的组件定义了模块，版本，发行信息好哥更新信息。（注意组件的顺序是重要的）

一个元素中的发行信息可能是一个cpe名称的歧义来源。因为有很多种办法去表达公司和组织的名称，对于cpe，使用的名称是最具有代表性的组织信息或DNS名称，即使domain的名字不同于公司名称，仍然推荐使用主机名作为cpe的名称，下面展示了几个代表性的示例。

Origanization Full Name DNS Domain CPE component
Cisco Systems, Inc. cisco.com cisco
The Mozilla Foundation mozilla.org mozilla
University of Oxford oxford.ac.uk oxford

cpe名称是大小写不敏感的， 为了减少歧义，所有的cpe名称都应该小写。

空的组件是合法的，他们指定平台名称的一部分，其中的任何方面该平台是有效的。例如下面的cpe名字指定了所有的安装了service pack 4的Microsoft Windows2000版本。

cpe://microsoft:windows:2000::sp4

缩略语

一些单词和短语在it产品设计中出现频繁，cpe名称中包含这些通常词汇将会很有优势。还会让cpe名称紧凑易于使用。为了有助于简化一些长的cpe名称，在适当的时候简化缩写，但是一个特别的缩写可以被应用于多个名字，例如，service pack和support pack的家还都是sp，但是只能有一个缩写允许用于这两个名字。

下方的表格罗列了被应用于cpe名称组件中的多词组合缩写。

Full Designation CPE Abbreviation Remarks
version 3.4 3.4 版本号有特殊语义用于匹配
release 3 r3 供应商的发布版本以此种形式进行缩写
release candidate 2 rc2 开源软件中的候选版本很常见。
service pack 4 sp4 服务包的缩写
support pack 2 sptp2 support的缩写为spt
home edition home 出现在后面的edition应该被抛弃
professional edition pro 专业版
service release 2 sr2 服务发布版本
security rollup sr 这个缩写似乎专用于微软
advanced server advanced 出现在后面的server应该被抛弃
standard edition std 啊哈

除非知名的官方多词产品名，否则应该使用全拼，下划线代替空格。下面的例子展示的是Zone Labs ZoneAlarm Internet Security Suite version 7.0。

cpe:///zonelabs:zonealarm_internet_security_suite:7.0

除了附录部分的一些词，单个单词的不应该缩写。

禁用字符

  • colon（:）,冒号用于在名称元素中分隔组件
  • semicolon(;)，分号用于在组件部分中分隔名称元素
  • slash(/), 斜线用于分隔cpe名称部分。
  • percent-sign(%)，用于uri中的字符编码。

硬件部分

硬件部分由一个或多个组件构成，规则定义了开头部分的三个组件，其他的可能依次添加在之后。

{vendor}:{family}:{model}

第一个组件必须是硬件元素的供应商。应该用公司的DNS的domain name作为组织标签。

第二个部分是硬件的家族描述，很多供应商定义了产品线和家族名使用一个有特色的名称。假如一个特别的产品没有这样的名字，留空。

第三个组件应该被用于硬件模块的设计，大多数硬件发行商会使用内部一致的型号系统。

示例

下面的示例定义了一个特别的笔记本电脑硬件平台，供应商是戴尔电脑，产品线名称是Inspiron，模块编号为8500.

cpe:/dell:inspiron:8500

多个硬件元素相关时，多硬件元素会被使用。下面的示例中的cpe名称定义了一个有着加密加速器模块的网络路由，注意元素的顺序不重要。

cpe:/juniper:m-series:m7i;juniper:es-pic

硬件元素也可以用于定义虚拟硬件，在下面示例中，虚拟硬件平台作为类似于实际硬件平台充当同样的角色。

cpe:/emc:vmware:esx:2.5

操作系统部分

操作系统部分包含一个或多个组件，意味着前三个组件被规范定义，别的组件紧随其后，操作系统用别的名字定义是困难的，因为每个发行商遵循着不同的scheme为他们的产品发行版本命名。

{vendor}:{family}:{version}

第一个组件必须是操作系统的发行商。

第二个组件必须是操作系统的家族名称或者通用名称。

第三个组件必须是操作系统版本号或者主版本定义。

此外组件也能被用于特别描述一个个人的操作系统，因为和每种类型的操作系统有所区别，这些组件在cpe中可能未定义。例如一些操作系统可能在不同的发行版本中区分，其他一些有更新的级别和子版本的发布。组件可能也被用于定义一个特殊语言版本的一个操作系统。

示例

下面的示例定义了一个微软的windows xp操作系统，专业版，更新级别为Service pack 2.

cpe://microsoft:windows:xp:pro:sp2

这个示例定义了Apple的Tiger MacOS X 发行版本4

cpe://apple:macos:10.4

Fedora项目 “Fedora Core”Linux分发的 release 6

cpe://fedoraproject:fedora_core:6

下方定义了Microsoft Windows2000，更新等级为Serive Pack4

cpe://microsoft:windows:2000::sp4

应用程序部分

应用程序部分的元素由两个或多个组件所组成，意味着前四个组件由规范所定义，其他的组件可以被增加在其后。

{vendor}:{name}:{edition}

第一部分必须为应用程序、服务器、或软件包的提供商或者发行商，如果应用程序是供应商 A 对开发人员 B 创建的系统的扩展。（In cases where an application is an extension by vendor A of a system created by developer B，说真的我每个字都认识，连起来就是看不懂）供应商A的域名应该被用于第一个组件的基础部分。

第二部分必须是应用程序的产品家族名称或者特别的产品名，一些小的组织会使用一些相同的组织名和产品名，在这种情况下，名称应该在每个组件中都出现。

第三部分必须是一个版本名称、版本号、发行版本或者更新级别。

其他的组件部分可能被用于描述和个人应用相关的特殊版本。

示例

定义一个JBoss 4.04 应用服务器。

cpe:///jboss:joboss:4.0.4

定义一个Oracle数据库10g，发行版本2.

cpe:///oracle:database:10g:r2

定义一个Adobe Acrobat 6.0文档生成应用程序，标准版或专业版。

cpe:///adobe:acrobat:6.0:std!pro

总结

CPE的语法格式是一种特殊的字符串，用于确定特定软件产品的版本。语法格式如下：

cpe:2.3:a:vendor:product:version:update:edition:language:sw_edition:target_sw:target_hw:other

其中每一个字段的含义如下：

  • 2.3：表示CPE语法的版本。
  • a：表示产品的类型，a代表应用程序，h代表操作系统，o代表其他。
  • vendor：表示软件的供应商，例如Microsoft或Adobe。
  • product：表示软件的名称，例如Windows或Acrobat。
  • version：表示软件的版本号，例如7.0或11.0.1。
  • update：表示软件的更新版本，例如Service Pack 1。
  • edition：表示软件的版本，例如Professional或Enterprise。
  • language：表示软件的语言，例如English或German。
  • sw_edition：表示软件的版本，例如Standard或Premium。
  • target_sw：表示软件的目标软件平台，例如Windows或Mac OS。
  • target_hw：表示软件的目标硬件平台，例如x86或x64。
  • other：表示其他任何有关软件的信息。

需要注意的是，不是所有字段都需要提供，只需要提供足以确定软件版本的字段。例如，一个简单的CPE字符串可能只包含vendor，product和version字段。
