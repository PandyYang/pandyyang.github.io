testing.t

testing包提供了Go package的自动单元测试支持，其旨在与go test命令一起使用，它会自动执行以下形式：

func TestXxx(*testing.T)

注意Xxx不能以小写字母开头，这个函数名提供了一个测试例程。

在这些函数中，使用Error， Fail或者相关方法来指示失败。

写一个单元测试的步骤：函数以Testxxx开头，文件名以_test结尾，创建一个文件，当执行go build命令时，其会自动被build指令排除，但是当执行go test时，其会自动被包含。

测试文件可以与被测文件在同一个包中，也可以在对应的后缀为“_test”的包中。

package abs

import "testing"

func TestAbs(t *testing.T) {
    got := Abs(-1)
    if got != 1 {
        t.Errorf("Abs(-1) = %d; want 1", got)
    }
}

假如测试文件在同一个包中，它将不可导出。即便是大写开头。（go中规定大写开头的函数才可以被导出）

package abs_test

import (
    "testing"

    "path_to_pkg/abs"
)

func TestAbs(t *testing.T) {
    got := abs.Abs(-1)
    if got != 1 {
        t.Errorf("Abs(-1) = %d; want 1", got)
    }
}

testing.b

函数是这种形式：

func BenchmarkXxx(*testing.B)

当执行go test时加上-bench标志，Benchmarks会按照顺序执行。

一个简单的benchmark函数看起来是这样。

func BenchmarkRandInt(b *testing.B) {
    for i := 0; i < b.N; i++ {
        rand.Int()
    }
}

这个benchmark函数必须运行目标代码b.N次，在基准执行期间，调整 b.N 直到基准函数持续足够长的时间以可靠地计时。

输出

BenchmarkRandInt-8   	68453040	        17.8 ns/op

意味着BenchmarkRandInt循环执行68453040次，每次耗时17.8ns。

假如benchmark在运行之前要进行一些昂贵的步骤，timer可能需要被重新设置。

func BenchmarkBigLen(b *testing.B) {
    big := NewBig()
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        big.Len()
    }
}

benchmark需要并行执行时，需要使用RunParallel帮助函数，这种方式应该需要加上-cpu标志。

func BenchmarkTemplateParallel(b *testing.B) {
    templ := template.Must(template.New("test").Parse("Hello, {{.}}!"))
    b.RunParallel(func(pb *testing.PB) {
        var buf bytes.Buffer
        for pb.Next() {
            buf.Reset()
            templ.Execute(&buf, "World")
        }
    })
}

一个计算hash效率的简单示例：

package main

import (
    "crypto/md5"
    "testing"

    "github.com/spaolacci/murmur3"
)

var str = []byte("0804633fd8839931a8bd1ba461d4c74db2d05020")


func BenchmarkMD5(b *testing.B) {
    b.ResetTimer()

    for i := 0; i < b.N; i++ {
        md5.Sum(str)
    }
}

func BenchmarkMurmur128(b *testing.B) {
    h := murmur3.New128()
    b.ResetTimer()

    for i := 0; i < b.N; i++ {
        h.Reset()
        h.Write(str)
        h.Sum(nil)
    }
}

func BenchmarkMurmur64(b *testing.B) {
    h := murmur3.New64()
    b.ResetTimer()

    for i := 0; i < b.N; i++ {
        h.Reset()
        h.Write(str)
        h.Sum(nil)
    }
}

命令

go test -bench=. -benchmem

输出

goos: linux
goarch: amd64
pkg: ntisdk/test/hash/speed
cpu: Intel(R) Xeon(R) CPU E5-26xx v4
BenchmarkMD5-2           7997414               150.1 ns/op             0 B/op          0 allocs/op
BenchmarkMurmur128-2    15702196                77.28 ns/op           16 B/op          1 allocs/op
BenchmarkMurmur64-2     17242051                68.32 ns/op            8 B/op          1 allocs/op
PASS
ok      ntisdk/test/hash/speed  3.906s

名令的参数含义

参数 含义
-bench regexp 性能测试，支持表达式对测试函数进行筛选。
-bench . 则是对所有的benchmark函数测试，指定名称则只执行具体测试方法而不是全部
-benchmem 性能测试的时候显示测试函数的内存分配的统计信息
－count n 运行测试和性能多少此，默认一次
-run regexp 只运行特定的测试函数， 比如-run ABC只测试函数名中包含ABC的测试函数
-timeout t 测试时间如果超过t, panic,默认10分钟
-v 显示测试的详细信息，也会把Log、Logf方法的日志显示出来

输出的结果含义

结果项 含义
BenchmarkMD5-2 Benchmark_test 是测试的函数名
-12 表示GOMAXPROCS（线程数）的值为12
7997414 循环次数即b.N的值
150.1 ns/op 表示平均每次操作花费了150.1纳秒
0 B/op 表示每次操作申请了0Byte的内存申请
0 allocs/op 表示每次操作申请了0次内存

testing.f

“go test”和测试包支持模糊测试，这是一种测试技术，其中使用随机生成的输入调用函数以查找单元测试未预料到的错误。

例如：

func FuzzHex(f *testing.F) {
  for _, seed := range [][]byte{{}, {0}, {9}, {0xa}, {0xf}, {1, 2, 3, 4}} {
    f.Add(seed)
  }
  f.Fuzz(func(t *testing.T, in []byte) {
    enc := hex.EncodeToString(in)
    out, err := hex.DecodeString(enc)
    if err != nil {
      t.Fatalf("%v: decode: %v", in, err)
    }
    if !bytes.Equal(in, out) {
      t.Fatalf("%v: not equal after round trip: %v", in, out)
    }
  })
}

模糊测试维护一个种子语料库或一组默认运行的输入，并且可以为输入生成种子。可以通过调用 (*F).Add 或通过将文件存储在包含模糊测试的包内的目录 testdata/fuzz/（其中 是模糊测试的名称）中来注册种子输入。种子输入是可选的，但是当提供一组具有良好代码覆盖率的小种子输入时，模糊引擎可能会更有效地发现错误。这些种子输入还可以用作通过模糊测试识别的错误的回归测试。

模糊测试中传递给 (*F).Fuzz 的函数被视为模糊目标。模糊测试目标必须接受 *T 参数，后跟一个或多个随机输入参数。传递给 (*F).Add 的参数类型必须与这些参数的类型相同。模糊测试目标可能会发出与测试相同的方式发现问题的信号：通过调用 T.Fail（或调用它的任何方法，如 T.Error 或 T.Fatal）或通过panic。

启用模糊测试时（通过将 -fuzz 标志设置为与特定模糊测试匹配的正则表达式），将使用通过重复对种子输入进行随机更改而生成的参数调用模糊测试目标。在支持的平台上，“go test”使用模糊覆盖检测编译测试可执行文件。模糊引擎使用该工具来查找和缓存扩大覆盖范围的输入，从而增加发现错误的可能性。如果给定输入的模糊测试目标失败，则模糊测试引擎会将导致失败的输入写入包目录中目录 testdata/fuzz/ 中的文件。该文件稍后用作种子输入。如果无法在该位置写入文件（例如，因为该目录是只读的），

禁用模糊测试时，将使用在 F.Add 中注册的种子输入和来自 testdata/fuzz/ 的种子输入调用模糊测试目标。在这种模式下，模糊测试的行为很像常规测试，子测试以 F.Fuzz 而不是 T.Run 开始。

skipping

test或者benchmark可以在运行时跳过测试。

例如

func TestTimeConsuming(t *testing.T) {
    if testing.Short() {
        t.Skip("skipping test in short mode.")
    }
    ...
}

skip可以用在一些错误的模糊测试结果中，但是不应该考虑一个错误的输入。

func FuzzJSONMarshaling(f *testing.F) {
    f.Fuzz(func(t *testing.T, b []byte) {
        var v interface{}
        if err := json.Unmarshal(b, &v); err != nil {
            t.Skip()
        }
        if _, err := json.Marshal(v); err != nil {
            t.Errorf("Marshal: %v", err)
        }
    })
}
