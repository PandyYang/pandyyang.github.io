如何使用Golang设计一个高效安全的Http服务
Golang web
技术
发布日期:   2023-03-14
作者:   Pandy
文章字数:   3.7k
阅读时长:   15 分

Golang有很多Web框架，例如Gin、Fasthttp以及Beego等著名开源框架。本文结合Fasthttp来研究下Golang如何处理http服务启动、运作、退出等优雅实现。其中可能会涉及到很多Go的高级知识，例如goroutine异步编程，waitGroup并发控制、Context线程通信、channel管道、single信号量等等概念，不要慌，好好学。（手动狗头～）

了解http库

http服务的本质是监听机器端口，收到请求，处理请求，返回请求。

类似于Java的RequestServlet和ResponseServlet，Golang也有其封装好的库，http.ResponseWriter, http.Request。

其中ResponseWriter用于Http handler组织Http响应。在源码层面，ResponseWriter只有三个方法。

  • Header() Header

    用于返回WriteHeader发送的header map。它是一个键为string，值为[]string的形式。

  • Write([]byte) (int error)

    Write方法会写入数据至Http连接中，以作为Http回应的一部分。说白了就是响应数据。将内容返回给前端。

  • WriteHeader(statusCode int)

    此方法会将状态码写入Http的Response header中。

而Request则表示服务器接收到的一个请求，或者从一个客户发送给另一个客户的请求。这样 看来，Response和Request是相对的，你是Request我就是Response，你是Response我就是Request。

Request相比Response有更多的参数和方法，假如你要和一个人讲话，首先你要认识这个人（不认识服务器怎么建立连接？），并且你要知道你是来干嘛的（Method, 对应增加、删除、修改、查询等等），你还要知道怎么去他所处的位置（URL，简单来说就是路径，官方就是统一资源定位符，因为互联网上的每个公开的数据，都有一个唯一的定位符来找到它），你还要带着一些必要信息，例如你参加某地区的圈子，你要有正确的你们约定好的会员身份信息（Proto，协议信息），还有你要会一些必要的接触信息，例如你说啥语言，你的爱好之类的(Header 请求头，表示服务器该如何处理、验证、提示、协商、管理一些信息的情况)，例如：

    Header = map[string][]string{
        "Accept-Encoding": {"gzip, deflate"},
        "Accept-Language": {"en-us"},
        "Foo": {"Bar", "two"},
    }

你会发现Header是个map，它自己好像也能传递一些信息，那我们就用Header传数据好了，你真是个大聪…。

Header的大小被服务器所限制，如果超过默认大小会出现错误：Request header is too large。所以人家的出现就不是为传递数据而生。有了头，肯定就有身体，所以Body必不可少。

你去见朋友干嘛？不就是憋了一肚子话吗？（Body 请求体）里面携带的就是你今天的脑袋里的，不对，肚子里的谈资。假如你除了一肚子话，还带了一副画想和朋友欣赏，你总不能直接拿着画过去吧，得用一些特殊的东西装起来（MultipartForm， 用于文件传输），除此之外你还带了些别的，可是我的读者暂时不感兴趣。

介绍完了这俩收发大佬，看个简单的例子吧。

package main

import (
    "fmt"
    "log"
    "net/http"
)

func handler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "Hi there, I love %s!", r.URL.Path[1:])
}

func home(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "这是主页")
}

func user(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "这是用户")
}

func createUser(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "这是创建用户")
}

func order(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "这是订单")
}

func main() {
    http.HandleFunc("/", home)
    http.HandleFunc("/user", user)
    http.HandleFunc("/user/create", createUser)
    http.HandleFunc("/order", order)
    log.Fatal(http.ListenAndServe("0.0.0.0:8080", nil))
}

启动此服务，你会发现很多值得思考的地方。

http.HandleFunc，接收了一个位置（即URL），还有一个匿名函数，即home， user等。它统一注册到http处，而http则位于一个地方（”0.0.0.0:8080”）。

log.Fatal接收一个error，会退出程序，而如果程序没有错，http.ListernAndServe不会返回error，则程序一直阻塞，监听着这个位置。说白了就是你去参加宴会，宴会一直没散场。来来往往的人觥筹交错，但是一旦发生突发情况，例如例行检查，群众惊慌了，返回一个error，则聚会散伙，程序结束。

URL在web框架中的处理，一般都是用的前缀树算法，想深入了解前缀树算法的奇妙之处，请参阅这篇文章：

https://geektutu.com/post/gee-day1.html

如果你真的理解了前缀树，并且稍微学会了Go如何使用结构体来组装、拆分一些各类请求问题，那http的使用核心，基本上就明白了。

学过面向对象的话，应该知道在使用一个对象前，要先进行初始化，Java的new即调用construct进行初始化的过程，go也有new，一般用于初始化一些容器，其实初始化就是分配内存，表象就是为一些变量设置初始值。结构体只是变量和方法的一种组织，我们用结构体只是为了方便构造对象。

一个http服务，想想应该包含哪些点？

  • 名称 （例如：文件下载服务）
  • 配置 （配置控制着程序的一些用户核心参数，例如最常见的监听端口）
  • 日志 （一个完善的程序必须要有运行日志，方便追踪监控程序状况、用户行为等等）

基本上有了这三个点，就能勉强写一个简单的http服务了。

我们尝试定义一个结构体。

type App struct {
    Name           string

配置

viper是一个第三方开源的配置管理的东西。简单来说就是你告诉他我的生日是啥时候，它会说2月30。

如何初始化一个Viper，具体参见github官方文档。我这里写一个简易版本的。我们用用到一个好玩的东西，sync.Once, 来确保类似于单例模式的 一个实现。因为配置是全局唯一，我们希望在多次调用new方法之后，都只有一个全局config。之后的logger也一样。

import (
    "fmt"
    "sync"

    "github.com/spf13/viper"
)

const (
    defaultConfigName = "MyConfig"
    ConfigPath        = "./"
)

var (
    defaultConfigInstance *viper.Viper
    onceConfig            sync.Once
)

type MyViper struct {
    viper.Viper
}

// NewConfig : Create new config provider
func NewConfig(name string) *viper.Viper {
    if defaultConfigInstance != nil {
        return defaultConfigInstance
    }

    if name == "" {
        name = defaultConfigName
    }

    // defer afterInit()

    c := viper.New()
    c.AutomaticEnv()
    c.SetConfigType("ini")
    c.AddConfigPath(ConfigPath)

    err := c.ReadInConfig()
    if err != nil {
        fmt.Printf("load config from %s failed, check the config file is exist. %s", ConfigPath, err.Error())
        return nil
    }

    // once可以防止内存可见性问题
    onceConfig.Do(func() {
        if defaultConfigInstance == nil {
            defaultConfigInstance = c
        }
    })

    return defaultConfigInstance
}

func GetConfig() *viper.Viper {
    if defaultConfigInstance != nil {
        return defaultConfigInstance
    }
    return NewConfig("")
}

日志

Logger是Golang自身的一个模块，但是要经过初始化，例如日志的形式、级别、所处位置等等。例如在下方，我定义了一个形式如：[MyLogger] - 2023/03/14 xxx.go:12: xxx的日志。

import (
    "fmt"
    "io"
    "log"
    "os"
    "sync"
)

const (
    defaultLoggerName = "MyLogger"
)

var (
    logPath               = ""
    defaultLoggerInstance *log.Logger
    onceLogger            sync.Once
)

func NewLogger(name string) *log.Logger {

    if defaultLoggerInstance != nil {
        return defaultLoggerInstance
    }
    logFilePath := GetConfig().GetString("common.LOG_PATH")
    logPath = logFilePath
    if !Exists(logFilePath) {
        CreateDirAndFile(logFilePath)
    }

    if name == "" {
        name = defaultLoggerName
    }

// 日志文件的创建类型 可读 可写 可追加
    f, err := os.OpenFile(logPath, os.O_RDWR|os.O_CREATE|os.O_APPEND, 0666)
    if err != nil {
        log.Fatalf("error opening file: %v", err)
    }

    multiWriter := io.MultiWriter(os.Stdout, f)
    sdkLogger := log.New(os.Stderr, fmt.Sprintf("[%s] - ", name), log.Ldate|log.Ltime|log.Lshortfile)
    sdkLogger.SetOutput(multiWriter)

    onceLogger.Do(func() {
        if defaultLoggerInstance == nil {
            defaultLoggerInstance = sdkLogger
        }
    })

    return defaultLoggerInstance
}

func GetLogger() *log.Logger {
    if defaultLoggerInstance != nil {
        return defaultLoggerInstance
    }

    return NewLogger("")
}

func SetLogger(logger *log.Logger) {
    if logger != nil && defaultConfigInstance == nil {
        defaultLoggerInstance = logger
    }
}

初始化完毕，要设置一些必要的信息，例如告诉服务，接收的参数最大为多少，如何解决跨域，要监听的路径有哪些，如果是加密传输，那么密钥在哪，等等一些必要的问题。

例如我们创建一个新的服务，传入的参数是服务启动地址，服务地址从配置文件中读取，例如：

addr := config.GetString("http.address")

func NewHTTPServer(addr string) *HTTPServer {
    f := &fasthttp.Server{
        MaxRequestBodySize: 64 * 1024 * 1024, //64MB
    }
    r := fasthttprouter.New()
    r.OPTIONS("/*_all", cors)

    server := &HTTPServer{
        addr:   addr,
        server: f,
        router: r,
    }
    return server
}

这个server我们可以定义为一个结构体，方便管理其中包含的很多关键信息

// HTTPServer : Fasthttp server
type HTTPServer struct {
    addr        string
    server      *fasthttp.Server
    router      *fasthttprouter.Router
}

里面的server是fasthttp的server。

其中的router是路由的意思，方便我们管理各个URL。

跨域

注意我们在上方的传入的一个cors，r.OPTIONS(“/*_all”, cors)。

OPTIONS 方法比较少见，该方法用于请求服务器告知其支持哪些其他的功能和方法。通过 OPTIONS 方法，可以询问服务器具体支持哪些方法，或者服务器会使用什么样的方法来处理一些特殊资源。可以说这是一个探测性的方法，客户端通过该方法可以在不访问服务器上实际资源的情况下就知道处理该资源的最优方式。

既然比较少见，什么情况下会使用这个方法呢？

最近在做跨域文件上传的时候，浏览器会自动发起一个 OPTIONS 方法到服务器。

func cors(ctx *fasthttp.RequestCtx) {
    SimpleCors(ctx)
    ctx.Response.Header.Set("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE, OPTIONS")
    ctx.Response.Header.Set("Access-Control-Allow-Headers", "Content-Type, Accept, X-Version,authorization")
    ctx.Response.Header.Set("Access-Control-Max-Age", "86400")
}

// SimpleCors : simple http cors header
func SimpleCors(ctx *fasthttp.RequestCtx) {
    ctx.Response.Header.Set(Cors, "*")
}

const (
    // Cors : http cors header
    Cors string = "Access-Control-Allow-Origin"
    // ContentType : http ContentType header
    ContentType string = "application/json"
)

核心是：ctx.Response.Header.Set(Cors, “*”)，Access-Control-Allow-Methods。都是一样的套路罢了。

路由

如果我们还想注册别的方法，好办也定义一些结构体和方法出来。

例如表示Route的结构体：

type HTTPRoute struct {
    Method      string
    Path        string
    Handler     fasthttp.RequestHandler
}

通过这个HttpRoute可以很方便地对方法、路径、已经对应的处理器进行管理。

如果要初始化，很简单：

var (
    httpRoutes = []*HTTPRoute{
        {Name: "query", Method: "get", Path: "/api/query", Handler: apiQuerySomtTing},
    }
)

我们可以定义一个HttpRoute的数组，来将所有URL统一管理。route最终都要注册进fasthttp的server *fasthttp.Server中，因此可以定义一个其和HttpServer绑定的函数：

func (s *HTTPServer) SetRoutes(routes []*HTTPRoute) error

httpServer.SetRoutes(httpRoutes)

在解析route的时候，注意需要根据方法的不同，来调用fastehttprouter的route方法依次处理，这里可以迭代数组，并且使用switch进行选择。

// SetRoutes : Set HTTP routes to server
func (s *HTTPServer) SetRoutes(routes []*HTTPRoute) error {
    if len(routes) == 0 {
        return fmt.Errorf("empty route list")
    }

    for _, route := range routes {
        if route.Path == "" || route.Handler == nil {
            continue
        }

        h := route.Handler
        uris := []string{route.Path}
        uris = append(uris, route.Aliases...)
        for _, uri := range uris {
            switch strings.ToLower(route.Method) {
            case "post":
                s.router.POST(uri, h)
            case "delete":
                s.router.DELETE(uri, h)
            case "put":
                s.router.PUT(uri, h)
            case "options":
                // BUGS(np) : CORS only
                s.router.OPTIONS(uri, h)
            case "patch":
                s.router.PATCH(uri, h)
            case "head":
                s.router.HEAD(uri, h)
            default:
                // Get for default
                s.router.GET(uri, h)
            }
        }
    }

至此，路由解析基本结束。

运行&停止

我们至此基本上将http的一些常见环节过了一遍。现在该启动了，但是有几个问题：

  1. http为什么在启动的时候，一直等待用户的请求而不中指程序呢？
  2. 用户可以在不停掉服务的情况下热更新配置文件吗？
  3. 面对用户的停止信号，和别的停止信号，程序怎么处理后续的一些问题呢？例如让没有处理完的请求，处理完，并且停止接收新请求，当所有请求结束时，执行释放内存等一些列后续操作
  4. 停止信号发出后，如果超过默认时间，需要强制关闭服务怎么做到？

关于问题1，我们可以用waitgroup做到这一点。它相当于Java的CountDownLatch，当收到所有信号，计数器减为0时，程序退出。看下代码吧：

func (app *App) Startup() {
    app.waiter.Add(1)
    
    // Signals
    sigKill := make(chan os.Signal, 1)
    signal.Notify(sigKill, syscall.SIGINT, syscall.SIGTERM)
    sigReload := make(chan os.Signal, 1)
    //signal.Notify(sigReload, syscall.SIGUSR2)

    go func() {
        for {
            select {
            case <-sigKill:
                // Shutdown clearly
                app.logger.Println("Exit signal captured")
                app.Shutdown()
            case <-sigReload:
                // Reload config
                app.logger.Println("Reload signal captured")
                app.config.ReLoadConfigFile()
            }
        }
    }()

    app.logger.Println("Application started")
    app.waiter.Wait()

    close(app.ShutdownSignal)
}

注意我们用到了带有缓冲区的channel来进行信号的捕获。

channel如果不带缓冲区：缺了任何一边都要阻塞另外一边。

channel带有缓冲区：放满阻塞生产者 空了阻塞消费者。

之后我们使用signal来通知其接收两种信号，一种是中断，一种是退出。

signal.Notify(sigKill, syscall.SIGINT, syscall.SIGTERM)

接收用户信号用于配置文件重载：

signal.Notify(sigReload, syscall.SIGUSR2)

定义go routine，在其中使用一个协程监听信号。

go func() {
        for {
            select {
            case <-sigKill:
                // Shutdown clearly
                app.logger.Println("Exit signal captured")
                app.Shutdown()
            case <-sigReload:
                // Reload config
                app.logger.Println("Reload signal captured")
                app.config.ReLoadConfigFile()
            }
        }
    }()

注意其中的select，其会将信号给获取出来。

当程序要停止时，调用shutdown方法：

func (app *App) Shutdown() {
    // 此处可以加上内存清理等一些操作
    defer func() {
        recover()
    }()

    app.waiter.Done()
    app.logger.Println("Application shutted")
}

shutdown中的waiter.Done() 会在程序做完清理工作时，放开主线程，程序终止。可能有人记得我们之前不是用了一个监听端口的操作，那个不是也能阻塞吗？

Golang规定，主线程退出时，所有子线程也要全部退出，所以我们在上方Start里用到了

defer s.server.Shutdown()

此defer无论是在用户终止，还是监听时报错，都能让fasthttp进行正确清理。

如果停止时希望让用户处理完请求怎么办？我们可以在捕获信号时，用一个Hook。例如：

给Hook假如一个超时机制，用户30s还未执行完，则强制退出。

for _, h := range hooks {
    ctx, cancel := context.WithTimeout(Context.Background(), time.Second * 30)
    err := h(ctx)
    cancel()
}
app.ShutDown()
