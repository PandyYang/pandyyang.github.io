如何使用Redis快速处理百万行文件
无标签
发布日期:   2023-04-23
作者:   Pandy
文章字数:   2.1k
阅读时长:   7 分

这篇文章涉及到的核心技术点。

  1. Go协程技术。用于操作文件的读取。

    “github.com/panjf2000/ants/v2”

  2. Redis线程池技术。用于Redis的多线程操作。

    “github.com/gomodule/redigo/redis”

  3. Linux的调优技术。主要是文件相关。

前言

考虑有这样一个场景，一个文件中包含了kv形式的json数据。k可能重复，我们希望将相同的k的v进行合并后，提取成一个新的文件，之后存入别的地方。

假如这个文件有1000w行。当你使用单线程进行处理时，你会发现，处理速度让人怀疑人生，特别是基于机械硬盘的处理。

其实很容易就想到用线程池等等技术去做这件事，但是需要注意，必须在保证内存足够低的情况下，高效且可靠地处理数据。这也许就是现实和理想之间的区别了。

如果要解决一个看似简单的问题，其实可能会花费一点心思的。

Go协程

我们的第一步想法是用一个流式读取的方式，去读大文件。go中的bufio中，有很多此类实现。例如

1. NewReader

file, err := os.Open(filePath)
baseReader := bufio.NewReader(file)
// 设置缓冲区大小 避免一行太长 加载被截断
reader := bufio.NewReaderSize(baseReader, 40960)

for {
            lineBytes, _, err := reader.ReadLine()
            if err == io.EOF {
                // 在此分支 文件读取完毕
                break
            }
        
        fmt.println(string(lineBytes))
    }

2. NewScanner

scanner := bufio.NewScanner(file)
for scanner.Scan() {
        lineBytes := scanner.Text()
        fmt.println(lineBytes)
}

两者的区别

  1. NewReader 是一个读取器，而 Scanner 是一个分词器

NewReader 是一个读取器，它将读取数据并将其缓存在内存中，以便进行更高效的读取操作。而 Scanner 则是一个分词器，它将输入分解为单独的令牌，并可以将令牌作为字符串、字节数组或自定义类型返回。

  1. Scanner 适用于按行读取的场景，而 NewReader 适用于更通用的场景

Scanner 最适合按行读取数据的场景，例如读取文件、网络连接或标准输入。Scanner 可以自动处理行结束符，并提供了方便的方法来处理每一行的数据。

而 NewReader 则适用于更通用的场景，例如读取任何类型的数据，包括二进制数据。NewReader 提供了更灵活的接口，可以读取任意长度的数据，并支持自定义的缓冲区大小和分隔符。

  1. Scanner 更容易使用，但可能不够灵活，而 NewReader 更灵活，但使用起来更复杂

Scanner 的接口非常简单，只需要调用 Scan 方法即可逐行读取数据，非常容易使用。但是，Scanner 仅适用于特定的数据格式，并可能无法满足所有需求。

相比之下，NewReader 更灵活，可以适用于多种数据格式和场景，但使用起来也更复杂，需要更多的代码来处理数据。

综上所述，NewReader 和 Scanner 都是逐行读取数据的工具，但它们的用途略有不同。如果您需要按行读取数据，则应使用 Scanner。如果您需要读取任意类型的数据或使用自定义缓冲区大小和分隔符，则应使用 NewReader。

在此处，我们用scanner进行文件处理。目前还没到文件分批处理的时候，例如将1000w行数据的文件，分为5个200w行的数据。这样处理速度会更快。

目前我的想法是：单线程读取，多线程处理。

例如：

    for scanner.Scan() {
        linestring := scanner.Text()
        
        go func(v string) {
            fmt.println(linestring)
        }(linestring)
    }

用一个协程可以实现此操作。但是协程的启动是非常快且多的， 假如我们直接将redis的操作放入go func，会直接导致连接占满，服务不可用。所以此时应该考虑，限制协程数量。

go的协程管理主要有，waitGroup(底层是原子计数)，channel&&select（生产消费者模式）等等。

我们今天要用一个封装好的第三方库，go ants，它的优点如下。

  • 自动调度海量的 goroutines，复用 goroutines
  • 定期清理过期的 goroutines，进一步节省资源
  • 提供了大量有用的接口：任务提交、获取运行中的 goroutine 数量、动态调整 Pool 大小、释放 Pool、重启 Pool
  • 优雅处理 panic，防止程序崩溃
  • 资源复用，极大节省内存使用量；在大规模批量并发任务场景下比原生 goroutine 并发具有更高的性能
  • 非阻塞机制

实际上就是，我们只需要在程序开始之间初始化好一些线程，例如100个，之后就不需要关心线程管理的诸多问题了。

说白了就是限制下线程数量。

解释：

Go 调度器工作时会维护两种用来保存 G 的任务队列：一种是一个 Global 任务队列，一种是每个 P 维护的 Local 任务队列。 当通过 go 关键字创建一个新的 goroutine 的时候，它会优先被放入 P 的本地队列。为了运行 goroutine，M 需要持有（绑定）一个 P，接着 M 会启动一个 OS 线程，循环从 P 的本地队列里取出一个 goroutine 并执行。当然还有上文提及的 work-stealing 调度算法：当 M 执行完了当前 P 的 Local 队列里的所有 G 后，P 也不会就这么在那躺尸啥都不干，它会先尝试从 Global 队列寻找 G 来执行，如果 Global 队列为空，它会随机挑选另外一个 P，从它的队列里中拿走一半的 G 到自己的队列中执行。

Redis线程池

虽然实际证明Redis的单线程具有更快的操作优势，但是线程池主要处理多客户端连接等等问题。

我们用sync.Once处理单例模式。这在初始化日志、配置、线程池、内存数据库等等场景下都适用。

func NewRedisPool() *redis.Pool {

    if redisPool != nil {
        return redisPool
    }

    onceRedis.Do(func() {
        if redisPool == nil {
            pool := &redis.Pool{
                MaxIdle:         10000,
                MaxActive:       0, // max number of connections
                IdleTimeout:     240 * time.Second,
                Wait:            true,
                MaxConnLifetime: 600 * time.Second,
                Dial: func() (redis.Conn, error) {
                    config := GetConfig()
                    netWork := config.GetString("redis.NETWORK")
                    host := config.GetString("redis.HOST")
                    port := config.GetString("redis.PORT")

                    if netWork == "" || host == "" || port == "" {
                        GetLogger().Printf("connect redis failed, check config failed, network %s, host %s, port %s", netWork, host, port)
                        return nil, errors.New("connect redis failed, redis config must not be null")
                    }

                    c, err := redis.Dial(netWork, fmt.Sprintf("%s:%s", host, port))
                    // u can choose the redis db use method blow
                    chosenDB := config.GetInt("redis.DB")
                    if chosenDB != 0 {
                        _, err = c.Do("select", chosenDB)
                    }
                    if err != nil {
                        GetLogger().Printf("connect redis failed, %s", err.Error())
                        return nil, err
                    }
                    return c, nil
                },
            }
            redisPool = pool
        }
    })

    return redisPool
}

Redis理论上可以跑满100%核心，当后台有aof或rdb任务时，可能会抽调一个cpu核心，去执行持久化操作。

Redis的持久化和过期机制可以放置内存占用过大。我用过很多内存数据库，发现redis的rw效率，多进程操作效率是非常高的。很多内存数据库是不支持多进程的，会直接加上锁，进行进程独占，为了避免很多麻烦。因为其在内存中可能构造lsm树，在衡量内存和性能之间的一个权衡之举吧，这个我可以理解的。

如果查询数据量巨大，可以引入布隆过滤器，此切面会提高极大性能的同时，只占用很少的内存，然而那些很低的误报率，就看个人如何选择了。

Redis的性能，和网络以及机器的配置相关。网络带宽大，机器主频高，会显著加快处理速度。但是有时候客户的机器不好改一些参数，例如ulimit，系统限制的打开文件数，这个就很尴尬了。所以尽量调一个好的参数出来吧。

Redigo在进程并发操作时，会出现问题，官方给出的方案如下：

    redisCli := redisPool.Get()
    defer redisCli.Close()

    res, err := redisCli.Do("get", name)

将DB的get和close在一个协程之间进行处理掉。否则会引发并发错误。具体细节情Google吧。
